Cross-Cutting Notes Verification

Context & Problem Statement

Customers onboarding to Imp Wizard — a beverage-agnostic platform serving breweries, wineries, cideries, distilleries, kombucha producers, coffee roasters, soda makers, and more — currently complete raw import spreadsheets manually. This creates friction, import errors, and slow time-to-value.

This project replaces that experience with a guided, step-by-step implementation wizard where:

  • Salesforce Opportunity/Account data auto-populates the customer profile
  • QuickBooks exports can be uploaded, mapped, and imported to supplement customer/contact data
  • Products and Package Types can be imported from any structured spreadsheet via a flexible column-mapping uploader
  • Customers or implementation specialists search a 14,000-item catalog classified by producer type
  • Items are staged, edited inline, validated, and exported as clean .xlsx files for import
  • A full changelog/audit log records every action taken by customers and implementation specialists
  • The tool remains useful after implementation — customers can bulk import new items from suppliers at any time

Salesforce integration is bidirectional: pull customer/project data in at the start, push status and export events back out at key milestones.

Key Architecture Decisions

Decision 1
Frontend approach
✓ ASP.NET Core 8 Web API + React (TypeScript/Vite)
Blazor Server was considered but ruled out due to its SignalR/WebSocket session dependency — customers filling out onboarding data over 30+ minutes cannot tolerate a dropped connection losing their state. React was chosen for its superior ecosystem for this UI pattern: dnd-kit (drag-and-drop), TanStack Query (server state), Zustand (local state), and react-dropzone (file uploads). The .NET backend handles all heavy lifting: file parsing, Salesforce REST calls, EF Core, Excel generation.
Decision 2
Database
✓ SQL Server with Full-Text Search
Natural fit for .NET / Azure. Full-Text Search (FTS) required to serve fast keyword queries against a 14,000-item catalog across ItemName, SearchAliases, and Description columns. Azure SQL is the future hosting target.
Decision 3
Salesforce integration scope
✓ Bidirectional — pull on email match + push status/export events
Email entered in Step 1 triggers an async SF Contact lookup. If a match + linked Opportunity is found, profile fields are pre-filled. All pre-filled fields remain editable. Status changes and export events are pushed back to SF. The NoOpSalesforceService runs when Enabled = false so the app works without SF credentials during development.
Decision 4
QuickBooks integration
✓ File upload (CSV / IIF) — no live API connection
Avoids QB OAuth app registration complexity. Customer or specialist exports data from QuickBooks, uploads the file, and the wizard parses/maps/flags it. Simpler, faster to implement, and works with both QB Desktop (IIF) and QB Online (CSV export).
Decision 5
Item catalog source
✓ CSV spreadsheet bulk seeder (14,000 items)
Items exist in spreadsheet form. A CatalogItemSeeder with upsert-on-name logic loads all items safely and can be re-run as the catalog grows. Items classified by producer type via a many-to-many join table (one item can belong to Brewery + Kombucha + Soda simultaneously).
Decision 6
Products & Package Types source
✓ Flexible uploader — any structured spreadsheet, any source
Products come from anywhere: supplier price lists, ERP exports, implementation templates. A column-mapping step lets users assign their file's columns to target fields. Mappings can be saved as named templates for reuse on future uploads. Works during implementation and post go-live.
Decision 7
Hosting
✓ Azure (future) — out of scope for MVP
Azure App Service + Azure SQL is the eventual hosting target. Architecture decisions (EF Core, ASP.NET Core, Vite build output) are all Azure-compatible. Hosting setup deferred until development is further along.

Cross-Cutting Notes

Beverage-Agnostic Design

  • No brewery-specific language in the UI — use "your products," "your items," "your setup"
  • Producer type drives catalog filtering, category lists, and item classification throughout the wizard
  • New producer types added by inserting a row in ProducerTypes — no schema change needed
  • Launch types: Brewery, Winery, Cidery, Distillery, Kombucha, Coffee, Soda, Other

Audit Log — Standing Requirement

  • Every data write (create, update, delete, import, export) must call IAuditService.LogAsync
  • Called after the DB write succeeds, outside any transaction (so log is never rolled back)
  • "Edit on behalf" actions attributed to the specialist, not the customer
  • Visible to all roles — customers see their project log; specialists and admins see all

Catalog Scale (14,000 Items)

  • SQL Server Full-Text Search required for acceptable search performance
  • Paginated results (default 25 per page)
  • CSV bulk seeder with upsert-on-name logic — safe to re-run as catalog grows
  • Admin can import/export CSV at any time for ongoing maintenance

Flexible Product/Item Importer

  • Accepts any structured spreadsheet — supplier price list, ERP export, implementation template
  • User-driven column mapping with named template save/reuse
  • Row-level confirm/skip before import
  • Works during implementation and post go-live (bulk import mode)

Post-Implementation Continuity

  • The tool doesn't end at go-live — IsPostImplementation = true unlocks bulk import mode
  • Supplier import batches tracked separately from original implementation data
  • Changelog covers all post-go-live activity too
